# 日本語歌声変換モデル 要求定義書

## 1. プロジェクト背景

### 現状
- RVC (Retrieval-based Voice Conversion) は英語話し声データで事前学習されたHuBERTを特徴量抽出器として使用
- HuBERT base モデルは英語 LibriSpeech 960h で学習されており、日本語の音素体系をカバーしていない
- 歌声特有の長母音・ビブラート・メリスマなどの表現が話し声ベースのパイプラインでは劣化する

### 目標
声優の歌声ボーカルデータ（数曲分）を使い、日本語の発音精度が高い歌声変換モデルを構築する。

---

## 2. 現行アーキテクチャの課題分析

### 2.1 HuBERT特徴量の言語バイアス

| 項目 | 現状 | 問題 |
|------|------|------|
| 学習データ | 英語 LibriSpeech 960h | 日本語の母音体系（あいうえお）や子音遷移を正確に捉えられない |
| 出力次元 | 768次元 (layer 12) | 最終層は英語タスクに最適化されており、日本語では中間層の方が汎用的な音響特徴を保持する可能性がある |
| 入力 | 16kHz 話し声 | 歌声の広い周波数帯域・ピッチ変動に対応していない |

### 2.1.1 利用可能な日本語SSLモデル（2025年最新調査）

| モデル | 学習データ | 出力次元 | ライセンス |
|--------|-----------|---------|-----------|
| imprt/kushinada-hubert-base | 日本語TV放送 62,215h | 768 | Apache 2.0 |
| reazon-research/japanese-wav2vec2-base | ReazonSpeech v2.0 35,000h+ | 768 | Apache 2.0 |
| rinna/japanese-hubert-base | ReazonSpeech v1 19,000h | 768 | Apache 2.0 |
| Spin V2 | ContentVec微調整 | 768 | - |

### 2.2 ピッチ処理の限界

| 項目 | 現状 | 問題 |
|------|------|------|
| F0範囲 | 50-1100 Hz | 歌声（特に高音域）では不足する場合がある |
| F0量子化 | 256レベル（mel尺度） | 歌声の微細なピッチ表現（ビブラート・しゃくり）が量子化ノイズで劣化 |
| RMVPE処理 | 16kHz固定 | 歌声の高次倍音情報が欠落 |

### 2.3 セグメント長の制約

| 項目 | 現状 | 問題 |
|------|------|------|
| 学習セグメント長 | 17280サンプル (0.36秒 @48kHz) | 歌声のフレーズ構造・長母音を学習するには短すぎる |
| 前処理チャンク | 3.7秒 + 0.3秒オーバーラップ | フレーズ単位の文脈が失われる |

### 2.4 RetrievalインデックスのMismatch

- FAISS検索はHuBERT空間上でk=8近傍を取得
- HuBERT空間が日本語を適切にエンコードできていない場合、検索結果自体が不適切になる
- 歌声データは話し声と分布が異なるため、同一インデックス空間での検索精度が低下

## 3. 要求仕様

### 3.1 機能要求

#### FR-1: 日本語対応特徴量抽出器の導入
- **概要**: HuBERT baseの代替として、日本語音声で学習された自己教師あり学習モデルを導入する
- **候補モデル**:
  - **imprt/kushinada-hubert-base（最有力候補）**: 日本語TV放送62,215hで学習。768次元出力でRVCと互換性が高く、大規模日本語データによる高品質な音響表現。Apache 2.0
  - **ContentVec (日本語fine-tuned)**: RVCとの互換性が高い。768次元出力を維持できる
  - **Spin V2**: ContentVecベースの微調整モデル。768次元。話者不変表現に優れる
  - **reazon-research/japanese-wav2vec2-base**: ReazonSpeech v2.0 35,000h+で学習。768次元。Apache 2.0
  - **rinna/japanese-hubert-base**: 日本語に特化。ReazonSpeech v1 19,000hで学習。768次元。Apache 2.0
  - **WavLM Large (多言語)**: 日本語を含む多言語データで学習済み。1024次元
  - **XLS-R (wav2vec 2.0 多言語)**: 128言語対応。1024次元
- **要件**:
  - 出力次元が変わる場合、TextEncoder入力層の次元を合わせる修正が必要
  - layer選択を固定値12からハイパーパラメータ化する
  - 複数layerの加重和（weighted sum）オプションを追加する

#### FR-2: 歌声対応ピッチ処理の拡張
- **概要**: 歌声特有のピッチ特性に対応する
- **要件**:
  - F0範囲を50-1600 Hzに拡張（ソプラノ対応）
  - F0量子化レベルを256→512に増加（微細表現の保存）
  - ビブラート・しゃくり（portamento）のF0パターンを保存する平滑化パラメータの調整
  - RMVPE入力を16kHz→24kHzに拡張することの検討

#### FR-3: 学習セグメント長の拡張
- **概要**: 歌声フレーズの文脈をより長く学習する
- **要件**:
  - セグメント長を17280→34560サンプル（0.72秒）以上に拡張
  - VRAM使用量とのトレードオフを検証
  - 前処理のチャンク長をフレーズ区切りベースに変更するオプション

#### FR-4: 歌声データ前処理パイプラインの追加
- **概要**: 歌声データに特化した前処理を追加する
- **要件**:
  - 歌声用Slicer（ブレス位置・フレーズ境界での分割）
  - 無音判定閾値の歌声向け調整（-42dB→-50dB等、残響を考慮）
  - ピッチ正規化オプション（キー統一のためのF0シフト前処理）
  - データ拡張: ピッチシフト（±2半音）、テンポ変更（±10%）、フォルマントシフト

#### FR-5: TextEncoder/Decoderの日本語最適化
- **概要**: モデル内部のエンコーダ・デコーダを日本語歌声に最適化する
- **要件**:
  - TextEncoderのattention head数・layer数を増やすオプション（2heads/6layers → 4heads/8layers）
  - フォルマント構造が日本語母音で正確に再現されることの検証
  - GeneratorNSFのハーモニクス生成が日本語歌声に適切か検証

#### FR-6: 多話者・歌声スタイル対応
- **概要**: 声優ごとの歌唱スタイルを学習できるようにする
- **要件**:
  - Speaker Embedding次元の拡張検討（256→512）
  - 歌唱スタイルの条件付け（ファルセット/地声/ミックスボイス等）の検討

### 3.2 非機能要求

#### NFR-1: 推論性能
- リアルタイム比1.0以上（CPU推論）/ 0.1以下（GPU推論）
- レイテンシ: ストリーミング推論で200ms以下

#### NFR-2: 学習データ効率
- 10分程度の歌声データで実用的な品質を達成
- データ拡張により実効的な学習データ量を3-5倍に拡張

#### NFR-3: 互換性
- 既存RVCモデル（v2 768次元）との後方互換性を維持するオプション
- 既存WebUIから新モデルを利用可能にする
- 既存の推論パイプライン（pipeline.py）を拡張する形で実装

#### NFR-4: 品質指標
- MOS (Mean Opinion Score): 4.0以上（5段階評価、歌声の自然さ）
- F0 RMSE: 20 cents以下（ピッチ精度）
- MCD (Mel Cepstral Distortion): 6.0 dB以下
- 日本語子音明瞭度: 聴取テストで90%以上

## 4. アーキテクチャ改修方針

### 4.1 Phase 1: 特徴量抽出器の置換（最優先）

```
[現行]
歌声WAV → 16kHz変換 → HuBERT(英語) → layer12固定 → 768次元特徴量

[改修後]
歌声WAV → 16kHz変換 → 日本語SSL※ → 加重layer選択 → N次元特徴量 → 次元調整層
                                                                    ↓
                                                              768次元特徴量
```
※SSL = Self-Supervised Learning モデル（最有力候補: imprt/kushinada-hubert-base, 日本語TV放送62,215hで学習済み）

**主な変更箇所**:
- `infer/lib/jit/get_hubert.py`: モデルローダーを抽象化し、複数バックエンド対応
- `infer/modules/vc/pipeline.py:191-199`: 特徴量抽出を差し替え可能に
- `infer/modules/train/extract_feature_print.py:114-129`: 学習用特徴量抽出
- `configs/v2/*.json`: `ssl_model`パラメータ追加

### 4.2 Phase 2: ピッチ処理の歌声対応

**主な変更箇所**:
- `infer/modules/vc/pipeline.py:88-162`: F0範囲・量子化の拡張
- `infer/lib/rmvpe.py`: 入力サンプルレート・周波数範囲の拡張
- `configs/v2/*.json`: ピッチ関連パラメータの追加

### 4.3 Phase 3: モデル構造の最適化

**主な変更箇所**:
- `infer/lib/infer_pack/models.py:565-721`: TextEncoder/Decoderパラメータ拡張
- `infer/lib/train/data_utils.py:85-88`: セグメント長拡張
- `infer/modules/train/train.py:464-468`: 歌声向け損失関数の調整

### 4.4 Phase 4: 前処理・データ拡張

**主な変更箇所**:
- `infer/modules/train/preprocess.py:77-101`: 歌声用前処理追加
- 新規: データ拡張モジュール

## 5. 学習データ要件

### 5.1 入力データ
- **フォーマット**: WAV, 48kHz, モノラル推奨
- **最小データ量**: 10分（3-5曲相当）
- **推奨データ量**: 30分以上（10曲以上）
- **内容**: ボーカルのみ（伴奏除去済み）
- **品質**: SNR 30dB以上、クリッピングなし

### 5.2 データ準備
- UVR5によるボーカル分離（既存機能を活用）
- 歌声区間の自動/手動セグメンテーション
- ピッチレンジの事前分析（声域確認）

## 6. 評価計画

### 6.1 客観評価
| 指標 | 目標値 | 測定方法 |
|------|--------|----------|
| F0 RMSE | < 20 cents | 変換音声 vs 原音のF0比較 |
| MCD | < 6.0 dB | メルケプストラム歪み |
| PESQ | > 3.5 | 音声品質客観評価 |
| CER（文字誤り率）| < 10% | 日本語ASRによる歌詞認識精度 |

### 6.2 主観評価
| 項目 | 目標 | 方法 |
|------|------|------|
| 自然さMOS | > 4.0 | 5段階リスニングテスト |
| 話者類似度 | > 4.0 | ABXテスト |
| 日本語明瞭度 | > 90% | 歌詞書き取りテスト |

## 7. 実装優先度

| 優先度 | 項目 | 根拠 |
|--------|------|------|
| **P0** | FR-1: 日本語SSL導入（最有力: kushinada-hubert-base） | 最大のボトルネック。HuBERT英語の言語バイアスが全体品質に波及。kushinada-hubert-baseは62,215hの日本語データで学習済みかつ768次元でRVC互換 |
| **P1** | FR-4: 歌声前処理 | データ品質は全学習結果に影響。早期に整備すべき |
| **P1** | FR-2: ピッチ拡張 | 歌声品質に直結。実装コストも比較的低い |
| **P2** | FR-3: セグメント長拡張 | 品質向上に貢献するがVRAM制約との調整が必要 |
| **P2** | FR-5: エンコーダ最適化 | 効果検証後に調整すべきハイパーパラメータ |
| **P3** | FR-6: スタイル対応 | 基本機能完成後の拡張 |

---

## 8. リスクと制約

| リスク | 影響度 | 対策 |
|--------|--------|------|
| 日本語SSLモデルの品質不足 | 高 | 複数候補を比較評価。最悪HuBERTのJapanese fine-tuningで対応 |
| 学習データ量の不足 | 高 | データ拡張の充実、few-shot学習手法の検討 |
| VRAM不足（セグメント長拡張時） | 中 | gradient checkpointing、mixed precisionの活用 |
| 推論速度の低下 | 中 | モデルサイズを監視し、必要に応じて蒸留・量子化 |
| 既存モデルとの互換性破壊 | 低 | 設定ファイルでのバージョン分離、移行パス提供 |
