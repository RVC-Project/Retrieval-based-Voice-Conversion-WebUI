# 改善提案: データ戦略 -- 拡張・前処理・事前学習

> 日本語歌声変換の品質向上に向けたデータパイプライン全体の最適化

---

## 1. 概要

本提案は、10分間の声優歌声データから高品質な日本語歌声変換モデルを構築するためのデータ戦略を定義する。現行RVCパイプラインの前処理は話し声を前提として設計されており（Slicer threshold=-42dB、セグメント長3.7秒、overlap=0.3秒）、歌声のロングトーン・ビブラート・ブレスパターンに対して最適化されていない。

本提案では以下の3軸で改善を行う:

1. **前処理の歌唱最適化**: Slicerパラメータ・ハイパスフィルタの歌声対応
2. **データ拡張による実効データ量の拡大**: 10分 → 40-100分相当（4-10倍）
3. **段階的転移学習**: 大規模日本語音声 → 日本語歌声 → ターゲット話者の3段階

これにより、現状の「10分データでの直接学習」と比較して、MCD 15-25%改善、Whisper CER 30-40%改善を目指す。

---

## 2. 技術的詳細

### 2.1 歌唱最適化前処理

現行の `infer/modules/train/preprocess.py` における `PreProcess` クラスのパラメータを歌声に最適化する。

#### 変更対象パラメータ一覧

| パラメータ | 現在値 | 提案値 | 変更理由 | 影響範囲 |
|-----------|--------|--------|---------|---------|
| `threshold` | -42 dB | **-38 dB** | 歌唱は発声レベルが高く、-42dBでは無声区間を取り込みすぎる。ブレス前後の不要な低レベル区間を除外 | `Slicer.__init__()` |
| `min_length` | 1500 ms | **2000 ms** | ロングノート（全音符=2秒@BPM120）を途中で分割しないための最小保証長 | `Slicer.__init__()` |
| `min_interval` | 400 ms | **300 ms** | 歌唱のブレス間隔は話し声のポーズより短い（200-400ms）。短いブレスでも分割点として検出 | `Slicer.__init__()` |
| `hop_size` | 15 ms | **10 ms** | RMS算出の時間分解能を向上。歌唱の急峻なダイナミクス変化を捕捉 | `Slicer.__init__()` |
| `max_sil_kept` | 500 ms | **300 ms** | 歌唱ブレスは話し声のポーズより短い。セグメント両端の無音を最小化 | `Slicer.__init__()` |
| `per` | 3.7 s | **5.0 s** | 歌唱フレーズは話し声の文より長い。サビ1フレーズ（4-8秒）を分断しない | `PreProcess.__init__()` |
| `overlap` | 0.3 s | **0.5 s** | セグメント境界のクロスフェード領域拡大。接合品質向上 | `PreProcess.__init__()` |
| ハイパスフィルタ `Wn` | 48 Hz | **40 Hz** | 男性低音域の歌唱基本周波数（C2=65Hz、B1=62Hz）を保護。48Hzカットでは低音の倍音構造が損傷 | `signal.butter()` |

#### 品質フィルタリング（新規追加）

前処理パイプラインに以下のフィルタリングステップを追加する:

| フィルタ | 閾値 | 計算方法 | 除外基準 |
|---------|------|---------|---------|
| SNR | > 20 dB | `10 * log10(signal_power / noise_power)` 冒頭0.5秒をノイズ基準 | 低SNRセグメント除外 |
| クリッピング検出 | 連続3サンプル以上 `abs(sample) > 0.99` | 波形ピーク分析 | クリッピング箇所を含むセグメント除外 |
| RMS範囲 | -40 dB < RMS < -3 dB | セグメント全体のRMS | 無音または過大入力を除外 |
| DC成分 | `abs(mean(signal)) < 0.01` | 直流バイアス検出 | DCオフセットのあるセグメントを補正または除外 |

### 2.2 データ拡張

#### 2.2.1 ピッチシフト拡張（最重要、4倍化）

```python
import librosa

shifts = [-4, -2, 2, 4]  # 半音単位
for n_steps in shifts:
    y_shifted = librosa.effects.pitch_shift(
        y, sr=sr, n_steps=n_steps,
        bins_per_octave=12,
        res_type='soxr_hq'  # 高品質リサンプリング
    )
```

| パラメータ | 値 | 根拠 |
|-----------|-----|------|
| シフト量 | -4, -2, +2, +4 半音 | SPA-SVC (Interspeech 2024) で有効性実証。6半音超はフォルマント破綻 |
| `bins_per_octave` | 12 | 標準12平均律 |
| `res_type` | `soxr_hq` | sox互換の高品質リサンプリング |
| 生成数 | 元データの4倍（4シフト） | 10分 → 50分（元+拡張） |

**注意事項**: ピッチシフトデータは HuBERT 特徴量抽出と F0 抽出の**前**に適用する。シフト後の音声に対して改めて特徴量を抽出することで、F0値とHuBERT特徴量の整合性を保証する。

#### 2.2.2 タイムストレッチ拡張（2倍化）

```python
rates = [0.9, 1.1]  # 速度変更倍率
for rate in rates:
    y_stretched = librosa.effects.time_stretch(y, rate=rate)
```

| パラメータ | 値 | 根拠 |
|-----------|-----|------|
| ストレッチ率 | 0.9x（10%減速）、1.1x（10%加速） | 10%以内はアーティファクトが知覚困難。0.8x以下はピッチ品質低下 |
| 生成数 | 元データの2倍 | 10分 → 30分（元+拡張） |

**注意事項**: タイムストレッチはピッチを変えずにテンポを変更するため、F0の値自体は保持されるが、フレーム単位の対応が変わる。ストレッチ後に改めてF0を再抽出する必要がある。

#### 2.2.3 SpecAugment（学習時オンライン適用）

```python
# data_utils.py の get_audio() 内で適用
import torchaudio.transforms as T

spec_augment = torch.nn.Sequential(
    T.FrequencyMasking(freq_mask_param=20),   # 最大20メルビン幅のマスク
    T.TimeMasking(time_mask_param=80),         # 最大80フレーム幅のマスク
)

# 50%の確率で適用
if random.random() < 0.5:
    spec = spec_augment(spec)
```

| パラメータ | 値 | 根拠 |
|-----------|-----|------|
| `freq_mask_param` | 20 | 128メルビンの約15%。歌声のフォルマント構造を部分的にマスク |
| `time_mask_param` | 80 | 480hop @48kHzで約0.8秒相当。ロングノートの一部をマスク |
| 適用確率 | 50% | 全バッチに適用すると学習が不安定化。半数で十分な正則化効果 |
| 適用箇所 | `TextAudioLoaderMultiNSFsid.get_audio()` | オンライン適用でストレージ不要 |

#### 2.2.4 TTS逆翻訳拡張（GPT-SoVITS活用、3-5倍化）

最も大きなデータ量増加が見込めるが、実装の複雑性も最大。

```
[Phase 1] 10分データ → GPT-SoVITSモデル初期学習（3-5分の参照音声で学習可能）
    ↓
[Phase 2] 日本語歌詞テキスト（J-POP歌詞100曲分等）→ GPT-SoVITSで合成歌声生成
    ↓
[Phase 3] 合成歌声 → 初期RVCモデルでターゲット声質に変換（声質一貫性向上）
    ↓
[Phase 4] 品質フィルタリング: UTMOS > 3.5 のサンプルのみ採用
    ↓
[Phase 5] 元データと合成データの混合（比率 1:2〜1:3）
```

| パラメータ | 値 | 根拠 |
|-----------|-----|------|
| GPT-SoVITS参照音声 | 3-5分（10分データの一部） | GPT-SoVITSの最小要件 |
| 合成テキスト量 | 100曲分の歌詞（約30-50分相当の音声生成） | 日本語歌詞の音素カバレッジ |
| UTMOSフィルタ閾値 | > 3.5 | SaruLab/UTMOS22による自然性スコア。3.5未満は知覚的品質不足 |
| 混合比率 | 元:合成 = 1:2〜1:3 | 合成データ比率が高すぎると声質が薄まる |
| 最終データ量 | 30-50分 | 元10分 + 合成20-40分 |

#### 2.2.5 拡張の組み合わせと最終データ量

| 拡張手法 | 倍率 | 適用順序 | 累積データ量 |
|---------|------|---------|------------|
| 元データ | 1x | - | 10分 |
| ピッチシフト | +4x | 前処理前 | 50分 |
| タイムストレッチ | +2x（元データのみに適用） | 前処理前 | 70分 |
| TTS逆翻訳 | +3x（元データベース） | 独立生成 | 100分 |
| SpecAugment | - | 学習時オンライン | ストレージ増加なし |

**合計: 10分 → 70-100分相当**（SpecAugment除く）

### 2.4 カリキュラム学習（Stage 3内）

Stage 3のファインチューニングを3フェーズに分割し、データの難易度に基づいて段階的に学習する。

| Phase | エポック | 使用データ | 学習率 | SpecAugment | 特記 |
|-------|---------|-----------|--------|-------------|------|
| 1: クリーンデータ | 1-100 | SNR上位50%のセグメントのみ | 1e-4 | 無効 | 基本声質の安定した獲得。高品質データで基盤構築 |
| 2: 全データ | 101-300 | 全セグメント（拡張データ含む） | 5e-5 | **有効**（50%適用） | データ拡張・正則化を併用して汎化性能獲得 |
| 3: 困難データ重点 | 301-500 | 損失値上位30%のセグメントを2倍サンプリング | 2e-5 | 有効（30%適用） | 弱点セグメントの重点補強。過学習注意 |

**Phase 3の損失ベースサンプリング実装**:
```python
# 各セグメントの直近の損失値を記録
segment_losses = {}  # {segment_id: loss_value}

# Phase 3でのサンプリング重み計算
loss_values = np.array(list(segment_losses.values()))
threshold = np.percentile(loss_values, 70)  # 上位30%
weights = np.where(loss_values > threshold, 2.0, 1.0)
weights /= weights.sum()
# WeightedRandomSamplerで使用
```

## 4. メリット

1. **実効データ量の大幅増加**: 10分 → 70-100分相当で、少量データの過学習問題を根本的に緩和。RVCコミュニティの経験則「データ(分) x エポック = 500-600」に対し、拡張後は100分 x 5-6エポックで到達可能

2. **段階的転移学習による日本語歌声の基盤獲得**: nadare氏のVCTK事前学習アプローチを日本語歌声に特化して拡張。英語話し声→ターゲットの直接ジャンプではなく、日本語話し声→日本語歌声→ターゲットの段階的適応により、日本語音素空間と歌唱特性の両方を効率的に学習

3. **ピッチシフトによるF0範囲の拡大**: 歌声の音域は話し声より広い（2オクターブ以上）。±4半音のシフトにより、学習データの音域カバレッジが実質8半音分拡大。SPA-SVC論文で有効性実証済み

4. **カリキュラム学習による学習効率の最適化**: 高品質データから始めて徐々に難しいデータを導入することで、初期段階での不安定な学習を回避。Phase 3の損失ベースサンプリングにより、モデルの弱点を集中的に補強

5. **SpecAugmentによるストレージ不要の正則化**: オンライン適用のためディスク容量を消費せず、過学習を約20%削減（原論文報告値）。歌声の特定の周波数帯域への過剰適合を防止

6. **品質フィルタリングによるデータ純度の保証**: SNRフィルタとクリッピング検出により、UVR5分離後の残留ノイズやアーティファクトを含むセグメントを自動除外。「ゴミを入れるとゴミが出る」問題を事前に排除

7. **歌唱最適化前処理によるセグメント品質向上**: ロングノートの分断防止（per=5.0s）、ブレス位置での適切な分割（min_interval=300ms）、低音域の保護（ハイパス40Hz）により、歌声の音楽的構造を維持したセグメンテーションを実現

8. **既存チェックポイントの活用可能性**: nadare氏のBOOTH配布チェックポイント（japanese-hubert-base対応済み）をStage 2の初期化に利用できる可能性があり、Stage 1をスキップできる場合がある

## 5. デメリット・リスク

1. **ピッチシフト・タイムストレッチのアーティファクト混入**: librosaのフェーズボコーダベースの処理は、特に子音部分でフェージングアーティファクトを生じる。4半音以上のシフトではフォルマント構造が不自然になり、モデルが不自然な発声パターンを学習するリスクがある。対策: シフト量を±4半音に制限し、拡張データの比率を元データの4倍以内に抑える

2. **TTS逆翻訳の声質劣化チェーン**: GPT-SoVITS → RVC の2段階変換により、各段階の誤差が累積する。特にGPT-SoVITSの10分データでの初期品質が低い場合、合成音声自体の品質が不十分となり、「低品質データで低品質モデルを学習する」悪循環に陥るリスクがある。対策: UTMOS閾値を3.5以上に設定し、厳格な品質フィルタリングを実施

3. **Stage 1-2の計算コスト**: JVS+MoeSpeech（約650時間）の事前学習にはマルチGPU環境で数日-1週間の計算時間が必要。個人開発者にとっては非現実的な場合がある。対策: nadare氏の既存チェックポイントの活用、またはStage 1をスキップしてStage 2から開始する軽量パスの提供

4. **データセットのライセンス制約**: 東北きりたん・GTSingerは研究用途限定。Stage 2で使用したモデルの商用利用可否が不明確。特にStage 2の重みがStage 3に引き継がれるため、研究用データの「汚染」がファインチューニング後のモデルにも及ぶ可能性がある。対策: 商用パス（パターンB）と研究パス（パターンA）を明確に分離

5. **カリキュラム学習のPhase遷移タイミング判断**: 固定エポック数でのPhase遷移は、データ量や品質によって最適点が異なる。早すぎるPhase 2への遷移は基本声質の未獲得、遅すぎる遷移は過学習を招く。対策: 検証損失のプラトー検出による動的Phase遷移の実装

6. **JVS-MuSiCの24kHz制約**: JVS-MuSiCは24kHz収録であり、RVCの48kHz設定（`configs/v2/48k.json`）とのサンプリングレート不一致が生じる。アップサンプリングにより高域情報が欠落した状態で学習が行われ、Stage 2の歌声表現が16kHz以上の帯域で劣化する可能性がある

7. **SpecAugmentの歌声固有の問題**: 周波数マスキングがビブラートのフォルマント変動と干渉する可能性がある。時間マスキングがロングノートの持続部分を過度にマスクし、ロングトーンの学習を阻害するリスクがある

8. **segment_size拡大のVRAM影響**: 17280→34560へのsegment_size倍増により、バッチあたりのVRAM消費が約2倍に増加。batch_size=4でもRTX 3080（10GB）では不足する可能性がある

### 9.2 他の改善提案との依存関係

```
[本提案: データ戦略]
    ↑ 独立して実装可能（前処理最適化・データ拡張部分）
    ↑ 日本語HuBERT提案と相乗効果（転移学習部分）
    │
    ├── 日本語HuBERT導入 → Stage 2-3の特徴量抽出精度が向上。HuBERT変更後にデータ再抽出が必要
    │     └── 候補: `imprt/kushinada-hubert-base`(62,215h), `rinna/japanese-hubert-base`(19,000h), `reazon-research/japanese-wav2vec2-base`(35,000h+)
    ├── 損失関数改善 → Stage 2-3の学習効率が向上。独立して適用可能
    ├── ボコーダ改善 → 独立。データ戦略とは直交する改善
    └── F0処理改善 → ピッチシフト拡張データのF0抽出精度に影響
```

### 9.3 実行時の前提条件

- 歌声データセット（JVS-MuSiC等）のダウンロードと利用規約への同意
- Stage 1実行時: マルチGPU環境（またはnadare氏チェックポイントの利用）
- TTS逆翻訳実行時: GPT-SoVITSの別途インストールと初期設定

## 10. 推奨実装順序

効果/工数比の高い順に段階的に実装する。各ステップは独立して効果検証可能。

```
Week 1: 前処理最適化（即効性が最も高い）
  ├── [1-1] Slicerパラメータ変更（threshold, min_length, min_interval, hop_size, max_sil_kept）
  ├── [1-2] per=5.0s, overlap=0.5s への変更
  ├── [1-3] ハイパスフィルタ 48Hz → 40Hz
  └── [1-4] 品質フィルタリング（SNR, クリッピング検出）の追加
  → 効果測定: MCD, Whisper CERのベースライン比較

Week 2: データ拡張（コスト対効果が最も良い）
  ├── [2-1] ピッチシフト拡張スクリプト作成・実行
  ├── [2-2] タイムストレッチ拡張スクリプト作成・実行
  └── [2-3] 拡張データでの学習・効果測定
  → 効果測定: 拡張あり/なしのA/B比較

Week 3: SpecAugment統合
  ├── [3-1] data_utils.pyへのSpecAugment組み込み
  ├── [3-2] 適用確率・マスクパラメータのチューニング
  └── [3-3] 過学習削減効果の検証
  → 効果測定: 学習曲線の比較（train/val loss）

Week 4-5: 転移学習（最大の品質向上が見込まれるが工数大）
  ├── [4-1] 既存チェックポイント（nadare氏）の適用テスト
  ├── [4-2] Stage 2用データセットの収集・前処理
  ├── [4-3] Stage 2の歌声事前学習実行
  ├── [4-4] Stage 3のカリキュラム学習実装
  └── [4-5] フルパイプラインでの品質検証
  → 効果測定: 全指標での最終評価

Week 6+（オプション）: TTS逆翻訳
  ├── [6-1] GPT-SoVITS環境構築・初期学習
  ├── [6-2] 合成音声生成・UTMOSフィルタリング
  └── [6-3] 混合データでの再学習・効果検証
  → 効果測定: TTS拡張あり/なしの比較
```

### 意思決定チェックポイント

| タイミング | 判断内容 | Go基準 | No-Go時の対応 |
|-----------|---------|--------|-------------|
| Week 1終了 | 前処理最適化の効果確認 | MCD 3%以上改善 | パラメータ再調整 |
| Week 2終了 | データ拡張の効果確認 | MCD 10%以上改善 | シフト量・ストレッチ率調整 |
| Week 3終了 | SpecAugmentの効果確認 | val lossの過学習削減 | パラメータ調整またはスキップ |
| Week 4開始前 | 転移学習の投資判断 | Week 1-3の累積改善が15%以上 | 前処理+拡張のみで運用 |
| Week 5終了 | フルパイプラインの品質評価 | MCD < 6.5dB, CER < 12% | ボトルネック分析・追加調査 |
