## 3. 各タスクの詳細

### 3-1. タスク4-1: F0量子化ビン拡大（256→512）

**工数**: 3日 | **GPU要件**: 事前学習再実行（RTX 4090 / Cloud GPU）

#### 技術概要

現在のRVCでは、F0（基本周波数）をメルスケール上で256ビンに量子化し、`TextEncoder`内の`nn.Embedding(256, hidden_channels)`で離散埋め込みとしてエンコードしている。256ビンの分解能では、歌声のビブラート（5-7Hz, 振幅50-100cents）やこぶし・しゃくりの微細なピッチ変動を表現するには粗すぎる。

512ビンに拡大することで、ビブラートの波形をより忠実に再現し、ピッチ情報の損失を軽減する。

#### 変更対象

**Embedding定義（256→512に変更）:**

| ファイル | 行 | 変更内容 |
|---------|-----|---------|
| `infer/lib/infer_pack/models.py` | L44 | `nn.Embedding(256, hidden_channels)` → `nn.Embedding(512, hidden_channels)` |
| `infer/lib/infer_pack/models_onnx.py` | L51 | 同上 |

**f0_bin定義（256→512に変更）:**

| ファイル | 行 | 変更内容 |
|---------|-----|---------|
| `infer/modules/train/extract/extract_f0_print.py` | L38 | `self.f0_bin = 256` → `self.f0_bin = 512` |
| `infer/modules/train/extract/extract_f0_rmvpe.py` | L36 | `self.f0_bin = 256` → `self.f0_bin = 512` |
| `infer/modules/train/extract/extract_f0_rmvpe_dml.py` | L34 | `self.f0_bin = 256` → `self.f0_bin = 512`（チケット原案で漏れていたファイル） |

**F0量子化処理（推論側） -- 254/255がハードコードされている箇所:**

以下のファイルでは `f0_bin` を参照せず `254`（= f0_bin - 2）と `255`（= f0_bin - 1）が直接ハードコードされている。512ビン化する場合は `510` と `511` に変更するか、`f0_bin` パラメータを参照するように修正が必要。

| ファイル | 行 | ハードコード箇所 |
|---------|-----|---------------|
| `infer/modules/vc/pipeline.py` | L159, L161 | `* 254 /` → `* (f0_bin - 2) /`、`> 255` → `> (f0_bin - 1)` |
| `infer/lib/rtrvc.py` | L204, L206 | 同上 |
| `tools/rvc_for_realtime.py` | L197, L199 | 同上 |
| `infer/lib/infer_pack/onnx_inference.py` | L126, L128 | 同上 |

**assertのハードコード（extract系ファイル）:**

以下のファイルでは `coarse_f0()` 内で `(self.f0_bin - 2)` を使って計算しているにもかかわらず、assert文は `<= 255` と `>= 1` がハードコードされている。`f0_bin` を512に変更した場合、assert条件も `<= 511` に更新が必要。

| ファイル | 行 | 変更内容 |
|---------|-----|---------|
| `infer/modules/train/extract/extract_f0_print.py` | L101 | `<= 255` → `<= self.f0_bin - 1` |
| `infer/modules/train/extract/extract_f0_rmvpe.py` | L64 | 同上 |
| `infer/modules/train/extract/extract_f0_rmvpe_dml.py` | L62 | 同上 |

#### 互換性への影響

- **事前学習モデルとの完全非互換**: `emb_pitch`のEmbedding次元が変わるため、M3-Bで作成した事前学習チェックポイントは使用不可。事前学習の全再実行が必須
- **設定ファイル**: `f0_bin`をconfigのmodelセクションにパラメータとして追加すべき（現在はハードコード）

#### 効果測定基準

- F0 RMSE: M3ベースライン比で10%以上改善
- ビブラートRate/Extent: 原音との乖離がM3比で縮小
- A/B比較: ビブラートの自然さが主観的に向上

#### リスク

- Embeddingサイズ増加によるVRAM増加は微小（256→512個の`hidden_channels`次元ベクトル = +192 * 256 * 4B = +192KB）
- 量子化ビンが増えることで、学習データが各ビンに薄く分散し、稀なピッチ域の学習が不安定になる可能性がある。事前学習の十分なデータ量で緩和

### 3-3. タスク4-7: 適応的index_rate

**工数**: 3-5日 | **GPU要件**: 現行と同等

#### 技術概要

現在の推論パイプラインでは、FAISSインデックス検索結果と元の特徴量を`index_rate`（固定スカラー値）で線形補間している。

**現行コード** (`infer/modules/vc/pipeline.py` L202-217):
```python
if not isinstance(index, type(None)) and not isinstance(big_npy, type(None)) and index_rate != 0:
    npy = feats[0].cpu().numpy()
    # ...
    score, ix = index.search(npy, k=8)
    weight = np.square(1 / score)
    weight /= weight.sum(axis=1, keepdims=True)
    npy = np.sum(big_npy[ix] * np.expand_dims(weight, axis=2), axis=1)
    # ...
    feats = torch.from_numpy(npy).unsqueeze(0).to(self.device) * index_rate + (1 - index_rate) * feats
```

問題点: 子音区間や無声区間では、FAISS検索の上位結果が不安定になりやすく、固定のindex_rateで検索結果を強く混合すると子音のアタック感が失われる。一方、母音や安定した有声区間ではindex_rateを高くして音色の安定性を確保したい。

#### 変更対象

| ファイル | 変更内容 |
|---------|---------|
| `infer/modules/vc/pipeline.py` L202-217 | フレーム単位の適応的index_rate計算ロジックを追加 |
| `infer-web.py` | UIに適応モードの切替オプションを追加（任意） |

#### 設計方針

```python
# 適応的index_rateの計算例
def compute_adaptive_index_rate(feats, scores, base_rate, f0=None):
    """
    フレーム単位でindex_rateを調整する。
    
    - 検索スコアが高い（距離が近い）フレーム → index_rate を引き上げ
    - 検索スコアが低い（距離が遠い）フレーム → index_rate を引き下げ
    - F0が0（無声区間） → index_rate を大幅に引き下げ
    """
    # 検索の信頼度に基づく重み（0.0-1.0）
    confidence = 1.0 / (1.0 + scores[:, 0])  # top-1 距離の逆数を正規化
    confidence = (confidence - confidence.min()) / (confidence.max() - confidence.min() + 1e-8)
    
    # F0情報による無声区間の検出
    if f0 is not None:
        voiced_mask = (f0 > 0).float()
        confidence = confidence * voiced_mask  # 無声区間はindex_rate→0
    
    adaptive_rate = base_rate * confidence  # フレーム単位のindex_rate
    return adaptive_rate
```

#### 効果測定基準

- 子音明瞭度: Whisper CERでの改善（特にサ行・タ行・カ行）
- 音色安定性: 話者類似度が低下しないことを確認
- A/B比較: 固定index_rate vs 適応index_rateの聴感テスト

#### リスク

- 適応ロジックの閾値チューニングに試行錯誤が必要
- リアルタイム変換時のオーバーヘッド: フレーム単位の条件分岐追加だが、FAISSの検索コストに比べると無視できる程度

### 3-4. タスク4-3: kNN-SVC加算合成（倍音強化）

**工数**: 3-4週間 | **GPU要件**: RAM 16GB+

#### 技術概要

kNN-VC（k-Nearest Neighbor Voice Conversion）の手法を応用し、FAISS検索で得られた近傍特徴量を現在の単純な線形補間ではなく、倍音構造を考慮した加算合成で統合する。

現在のパイプライン（`pipeline.py` L210-213）では、検索された上位8件の特徴量を距離の逆二乗重み付き平均で合成しているが、これは周波数帯域の情報を区別せず平均化するため、倍音のリッチさが失われる。

#### 変更対象

| ファイル | 変更内容 |
|---------|---------|
| `infer/modules/vc/pipeline.py` | 検索結果の合成ロジックを拡張 |
| 新規: `infer/modules/vc/knn_synthesis.py` | 加算合成モジュール |

#### 主要な作業

1. **倍音分析モジュール**: 検索された近傍特徴量の倍音構造を分析し、ターゲット音声の基本周波数に基づいて倍音を選択的に合成
2. **周波数帯域別の重み付け**: 基本周波数帯域と高次倍音帯域で異なる重み付けを適用
3. **時間的平滑化**: フレーム間の倍音構造の急変を防ぐ平滑化処理

#### 効果測定基準

- MCD: M3ベースラインからの改善
- 倍音構造の忠実度: スペクトログラムの高次倍音の再現性を視覚的に比較
- 主観評価: 音色のリッチさ・厚みの改善

#### リスク

- アルゴリズムの複雑性が高く、デバッグに時間がかかる可能性
- RAM 16GB+が必要（大きなインデックスの倍音分析のため）
- 推論速度への影響が大きい可能性があり、リアルタイム変換との両立が課題

### 3-5. タスク4-6: TTS逆翻訳データ拡張（GPT-SoVITS）

**工数**: 2-3週間 | **GPU要件**: RTX 3090+

#### 技術概要

ターゲット話者の10分程度の歌声データを、GPT-SoVITSなどのTTSモデルで「逆翻訳」し、実効的なデータ量を3-5倍に拡張する手法。具体的には以下の流れ:

1. ターゲット話者の音声でGPT-SoVITSの話者適応を実施
2. 日本語歌詞テキストからTTSで合成音声を生成
3. UTMOS > 3.5のフィルタリングで品質を担保
4. 合成音声をRVCのファインチューニングデータに追加

#### 変更対象

| ファイル | 変更内容 |
|---------|---------|
| 新規: `tools/data_augmentation/tts_backtranslation.py` | TTS逆翻訳パイプライン |
| 新規: `tools/data_augmentation/quality_filter.py` | UTMOS品質フィルタ |

#### 主要な作業

1. **GPT-SoVITS統合**: GPT-SoVITSのAPIまたはCLIラッパーを作成し、バッチ合成を自動化
2. **品質フィルタリング**: UTMOS（Unified Training with Mean Opinion Scores）でスコア3.5未満の合成音声を自動除外
3. **データ混合戦略**: 実データと合成データの混合比率の探索（推奨: 実データ30-50%）

#### 効果測定基準

- 実効データ量: 10分→30-50分相当への拡張
- 過学習指標: Validation Lossの安定化
- MOS: データ拡張ありvsなしでの主観評価比較

#### リスク

- TTS合成品質に依存するため、GPT-SoVITSの話者適応品質がボトルネックになる
- 合成データの分布が実データと乖離している場合、学習が不安定になる
- GPT-SoVITSの環境構築コストが追加で発生

#### 前提条件

- M2-Bで実装したデータ拡張スクリプト（ピッチシフト±4半音）が利用可能であること
- UTMOS推論環境がセットアップ済みであること

#### 主要な作業

1. **BigVGAN選択時**: AMPBlockの完全移植（SnakeBeta + AntiAlias + Periodic Conv を統合ブロック化）。M3-Bで個別導入した各コンポーネントをAMPBlockに統合
2. **HiFTNet選択時**: iSTFTベースのデコーダ設計。NSF（Neural Source Filter）との統合が研究課題
3. **事前学習の完全再実行**: アーキテクチャ全面変更のため、M2/M3のチェックポイントは再利用不可

M3-B完了後に以下を確認してから着手を判断:
- M3-Bの「部分導入」で高域品質が十分改善されたか → 改善済みならスキップ可能

**工数**: 4-6週間 | **GPU要件**: A100 40GB+推奨

#### 技術概要

RVCの出力波形の後段にShallow Diffusion（浅い拡散モデル）を追加し、波形のポリッシュ（品質向上）を行う。RVC本体は高速なVITSベース推論を維持し、拡散モデルは5-10ステップのみ実行して微細なアーティファクト除去と高域の改善を担当する。

#### 変更対象

| ファイル | 変更内容 |
|---------|---------|
| 新規: `infer/lib/infer_pack/diffusion/` | 拡散モデル定義 |
| `infer/modules/vc/pipeline.py` | 推論パイプラインの後段に拡散ステップを追加 |
| `infer/modules/train/train.py` | 拡散モデルの学習ループ追加 |

#### 主要な作業

1. **軽量拡散モデルの設計**: DiffSinger / CoMoSpeech の手法を参考に、5-10ステップで収束する浅い拡散プロセスを設計
2. **VITSとの統合**: RVCのGeneratorNSF出力をノイズ付加の開始点とし、完全なガウスノイズからではなく「浅い」ステップのみ実行
3. **Rectified Flow検討**: 拡散ではなくRectified Flowを使えば、2-4ステップで同等品質が達成可能（推論レイテンシ+50ms程度）

#### 効果測定基準

- UTMOS: +0.5-0.8の改善（3.2→3.8-4.0）
- MOS: 主観評価で明確な品質向上
- 推論レイテンシ: +300ms以下（Shallow Diffusion）、+50ms以下（Rectified Flow）

#### リスク

- A100クラスのGPUが必要で、ローカル環境（RTX 4070 Ti Super / RTX 4090）での学習が困難
- 推論速度が大幅に低下し、リアルタイム変換（`tools/rvc_for_realtime.py`）には使用不可
- 拡散モデルの学習安定性が課題。学習データ量（10分）では拡散モデルの汎化が不十分な可能性

#### 前提条件

- M3までの全最適化が完了し、VITSベースの品質が限界に近いと判断された場合にのみ実施
- Cloud GPU（A100以上）のアクセスが確保されていること

### 3-8. タスク4-8: GRPO強化学習

**工数**: 6-8週間 | **GPU要件**: 4xA100 40GB+

#### 技術概要

Group Relative Policy Optimization（GRPO）をRVCのGenerator学習に適用し、UTMOSスコアを直接最適化する強化学習アプローチ。LLM分野でのRLHF/DPOの成功をTTS/VCに応用する。

#### 変更対象

| ファイル | 変更内容 |
|---------|---------|
| 新規: `infer/modules/train/grpo_train.py` | GRPO学習ループ |
| 新規: `infer/modules/train/reward_model.py` | UTMOSベース報酬モデル |

#### 主要な作業

1. **報酬モデル**: UTMOSを報酬関数として使用。追加で話者類似度・CERを組み合わせた複合報酬を設計
2. **GRPO学習ループ**: Generatorの出力に対してグループ相対報酬を計算し、ポリシー勾配で最適化
3. **安定化技術**: KLペナルティ、報酬クリッピング、参照モデルの凍結

#### 効果測定基準

- UTMOS: 直接最適化対象。+0.3-0.5の改善を期待
- MOS: UTMOS最適化がMOSと相関するか検証
- 品質劣化チェック: 報酬ハッキング（UTMOS高スコアだが実際の品質は低い）の検出

#### リスク

- **極めて高い計算コスト**: 4xA100で数百GPU時間が必要
- 報酬ハッキングのリスク: UTMOSスコアは高いが聴感上は不自然な音声が生成される可能性
- 研究段階の手法であり、歌声VCでの実績が限定的
- 学習の不安定性: 強化学習特有の報酬崩壊・モード崩壊のリスク

#### 前提条件

- M4の他タスクが全て完了した後、さらなる品質向上が必要と判断された場合のみ実施
- Cloud GPU（4xA100以上）の長期アクセスが確保されていること

## 4. 推奨実施順序

M4タスクの実施順序は、Go/No-Go判定(3)の結果とボトルネック分析に基づいて決定する。以下は推奨順:

### Phase 4-A: 低コスト・高効果（Week 9-10, 約2週間）

| 順序 | タスク | 理由 |
|------|--------|------|
| 1 | 4-7 適応的index_rate | 現行GPU環境で完結。子音品質の即効的改善 |
| 2 | 4-2 Weighted Sum チューニング | M2-Aの実装をベースにチューニングのみ。低コスト |
| 3 | 4-1 F0量子化ビン拡大 | 事前学習再実行が必要だが、コード変更は最小限 |

### Phase 4-B: 中規模投資（Week 11-14, 約1ヶ月）

| 順序 | タスク | 理由 |
|------|--------|------|
| 4 | 4-6 TTS逆翻訳データ拡張 | データ量のボトルネック解消。学習安定性の根本改善 |
| 5 | 4-3 kNN-SVC加算合成 | 検索パイプラインの改善。4-7の成果を前提に構築 |

### Phase 4-C: 大規模投資（Week 15+, 数ヶ月）

| 順序 | タスク | 条件 |
|------|--------|------|
| 6 | 4-4 BigVGAN/HiFTNet移行 | M3-Bの「部分導入」で高域品質が不十分な場合のみ |
| 7 | 4-5 Shallow Diffusion | VITSベースの限界に達した場合のみ |
| 8 | 4-8 GRPO強化学習 | 全最適化完了後のファイナルポリッシュ |

### 依存関係

```
4-7 (適応的index_rate) ─────────────────────────┐
4-2 (Weighted Sum チューニング) ─── 独立 ──────────┤
4-1 (F0量子化ビン拡大) ── 事前学習再実行必要 ───────┤
                                               │
4-6 (TTS逆翻訳) ─── 独立 ────────────────────────┤
4-3 (kNN-SVC) ── 4-7の成果を前提に構築可能 ────────┤
                                               │
4-4 (BigVGAN/HiFTNet) ── 事前学習完全再実行 ──────┤
  └── 4-1と同時に事前学習すれば工数削減可          │
4-5 (Shallow Diffusion) ── 4-4の後段に追加 ───────┤
4-8 (GRPO) ── 全タスク完了後 ─────────────────────┘
```

> **最適化ポイント**: タスク4-1（F0ビン拡大）と4-4（BigVGAN移行）はいずれも事前学習の完全再実行を伴う。両方を実施する場合は、同一の事前学習セッションにまとめることで、GPU時間を大幅に削減できる。

## 6. 懸念事項

### 6-1. 全体リスク

| # | 懸念事項 | リスク | 対策 |
|---|---------|--------|------|
| C1 | **事前学習の繰り返しコスト**: タスク4-1と4-4が別々に事前学習を要求するとGPU時間が倍増 | 高 | 同一セッションで統合実行。タスク選定時にバンドル可能性を考慮 |
| C2 | **M3成果物との互換性喪失**: M4の大規模変更でM3チェックポイントが無駄になるリスク | 中 | Go/No-Go判定(3)でM3の品質が十分か慎重に判断し、不要な追加最適化を避ける |
| C3 | **Cloud GPUコスト**: A100クラスの利用が複数タスクで必要 | 中 | Phase 4-A（ローカルで完結するタスク）を先に実施し、効果を見極めてからCloud投資を判断 |
| C4 | **研究的要素の多さ**: タスク4-3, 4-5, 4-8は研究段階の手法を含み、確実な品質向上が保証されない | 高 | 各タスクに明確なGo/No-Go基準を設定し、プロトタイプ段階で効果が確認できない場合は早期中断 |

### 6-2. ロールバック戦略

- **Phase 4-A**: gitタグ `pre-m4a` を作成。適応的index_rateやWeighted Sumチューニングは推論パイプラインの変更のみのため、ロールバックは容易
- **Phase 4-B**: データ拡張は学習データの追加であり、元データの削除はしないためリスク低。kNN-SVC は新規モジュールのため無効化が容易
- **Phase 4-C**: 事前学習の完全再実行を伴うため、ロールバック時はM3-Bのチェックポイント（`f0G48k_snakebeta.pth`）に復帰。gitタグ `pre-m4c` で保護
